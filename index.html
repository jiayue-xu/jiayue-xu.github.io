Optimizing Data Movement for GPU-Based In-Situ Workflow Using GPUDirect RDMA

发表于 2023-10-26

GPUDirect

发表于 2023-10-08 更新于 2023-10-19

引言

GPUDirect RDMA技术实现了GPU和网络设备之间的数据通信，但是要求两个设备共享相同的PCIe Root Complex，并且需要对设备驱动程序进行一些简单的更改，以便在各种硬件设备上启用此功能。

要使用GPUDirect RDMA特性，需要使用英伟达Kepler及以上架构、Tesla或Quadro系列的GPU，CUDA版本至少大于5.0。

如何访问GPU地址空间

在网卡和GPU之间建立GPUDirect RDMA通信时，网卡和GPU都作为PCIe设备，它们通过物理地址去访问对方的数据。而从PCIe设备的角度来看，所有物理地址都是它们PCI BAR中的一部分，因此网卡向GPU的BAR地址发起读写的方式和它向主机内存发起读写操作是一样的。

通常，BAR窗口可以通过CPU的MMU映射到用户或内核地址空间，作为MMIO地址。因此，GPU的显存和网卡的内存都可以映射到驱动中，访问它们就像访问主机内存一样方便。然而，由于当前的操作系统没有足够的机制在驱动程序之间交换MMIO区域的数据，NVIDIA内核驱动程序就需要导出函数来执行必要的地址转换和映射工作来支持GPUDirect RDMA。

实现GPUDirect RDMA需要用户空间的CUDA库和NVIDIA内核驱动的支持。

  1. 用户空间的CUDA库提供的最重要的功能之一是利用了UVA特性，在应用程序的核函数中可以向访问主机内存一样访问GPU的内存地址。

  2. 在注册主机内存到网卡时，我们使用了Linux内核为我们提供的get_user_pages函数和put_pages函数获得虚拟地址到物理地址的转换关系，以及执行pin操作。同样，要支持GPUDirect RDMA，也需要做类似的操作，这个功能由NVIDIA Kernel driver提供。

GPUDirect RDMA 协议栈

  1. CUDA运行时库（用户态）

  2. CUDA驱动程序（内核）：很重要的一个函数nvdia_p2p_get_pages

  3. nv_peer_memory:也是一个内核模块，为IB设备注册一个peer设备(GPU)，把它的dma_map、get_pages等函数注册上去，其中get_pages函数最终调用的就是nvdia_p2p_get_pages，这个函数具体实现不知道的

MLNX-OFED-5.x

mlx为了支持GPUDirect RDMA，对IB驱动进行了很多修改，其中注册MR的地方需要支持注册peer(GPU)的内存，最后就需要调用到CUDA驱动程序export出来的nvdia_p2p_get_pages

单机多卡GPU通信

GPUDirect是英伟达开发的一项技术，可实现GPU与其他设备（例如网卡）之间的直接通信和数据传输，而不涉及CPU。传统上，当数据需要在GPU和另一个设备之间传输时，数据必须通过CPU，从而导致潜在的瓶颈并增加延时。使用GPUDirect，网卡可以直接读写GPU内存，减少不必要的内存消耗，减少CPU开销并降低延迟，从而显著提高性能。GPUDirect技术包括GPUDirect Storage、GPUDirect RDMA、GPUDirect P2P和GPUDirect 视频。

GPUDirect P2P

某些工作需要位于同一服务器中的两个或多个GPU之间进行数据交换，在没有GPUDirect P2P技术的情况下，来自GPU的数据将首先通过CPU和PCIe总线复制到主机固定的共享内存，然后数据通过CPU和PCIe总线从主机固定的共享内存复制到目标GPU，数据需要复制两次。

有了GPUDirect P2P通信技术后，将数据从源GPU赋值到同一节点中的另一个GPU不再需要将数据临时暂存到主机内存中。如果两个GPU连接到同一PCIe总线，GPUDirect P2P允许访问其相应的内存，而无需CPU参与。

多机多卡GPU通信

AL计算对算力需求巨大，多机多卡的计算是一个常态，多机间的通信是影响分布式训练的一个重要指标。在传统的TCP/IP网络通信中，数据发送方需要将数据进行多次内存拷贝，并经过一系列的网络协议的数据包处理工作；数据接收方在应用程序中处理数据前，也需要经过多次内存拷贝和一系列网络协议的数据包处理工作。经过这一系列的内存拷贝、数据包处理以及网络传输延时等，服务器间的通信时延往往在毫秒级别，不能满足多机多卡场景对于网络通信的需求。

GPUDirect RDMA结合了GPU加速计算和RDMA技术，实现了在GPU和RDMA网络设备之间直接进行数据传输和通信的能力。它允许GPU直接访问RDMA网络设备中的数据，无需通过主机内存。

英伟达的GPUDirect技术支持在GPU之间，以及GPU和其他设备之间进行高效通信。最早的GPUDirect版本于CUDA 4.0发布，这项技术将一块主机内存区域注册到网卡适配器和GPU设备，在注册内存之后，DMA引擎能正确访问这块内存。因此，GPUDirect技术的本质就是允许这两设备对同一块主机内存区域进行读写。当要把GPU的数据通过IB网络发出去时，利用GPUDirect技术可以避免不必要的数据拷贝。MPI库比如MVAPICH2目前就已经利用了GPUDirect特性。在最近的CUDA版本中，GPUDirect已经被扩展为允许第三方PCIe设备直接读写GPU设备的内存，被称为GPUDirect RDMA。

消息传递接口（MPI）是HPC领域并行应用的事实标准，MVAPICH2是MPI针对IB、10Gb以太网/iWARP和RoCE的开源实现。MVAPICH2通过标准的MPI语义统一了GPU和主机内存之间的数据移动操作。这个特性首次实现于[H. Wang, S. Potluri, M. Luo, A. Singh, S. Sur, and D. K. Panda. MVAPICH2-GPU: Optimized GPU to GPU Communication for InfiniBand Clusters. In Int’l Supercomputing Conference (ISC), 2011]。MVAPICH2利用CUDA 4.0提出的统一虚拟地址(UVA)特性实现这个功能。节点间GPU通信分为以下3个步骤：GPU到本地主机内存、通过IB的本地主机内存到远程主机内存、远程主机内存到远程目标GPU，MVAPICH2通过将这些步骤设计为3级流水线优化了节点间的GPU通信性能。这三个阶段的流水线显著提高了不同节点上gpu之间的通信性能。MVAPICH2还为进程间的节点内gpu到gpu通信提供了多种解决方案，使用共享内存和CUDA进程间通信。

本节集中讨论MVAPICH2中现有的节点间GPU到GPU的通信设计是如何实现的，以及怎么使用GDR改善性能。

A. MVAPICH2 GPUDirect RDMA (MV2-GDR)

针对不同大小的数据包，MPI库通常使用不同的通信策略。我们分析在涉及GDR和GPU设备缓存时，这些策略是如何被选择的。对于主机缓冲区之间的小消息，如果实际涉及的数据移动很小，则进程之间同步的开销可能会导致相当大的开销。因此，避免任何同步开销是小消息通信设计的关键考虑因素之一。在IB发送/接收语义中，发送方和接收方进程应该用缓冲区信息发布匹配的发送和接收请求，以便在源缓冲区和目标缓冲区之间直接传输数据。接收应该在发送操作之前发布。在IB RDMA语义中，一个进程同时提供源缓冲区和目标缓冲区，在启动数据之前，必须在进程之间交换缓冲区地址和注册信息。为了避免这些同步开销，MPI库使用eager协议，其中数据通过接收进程中预先确定的中间缓冲区进行分阶段处理。当接收进程可以自由地将数据从中间缓冲区复制到最终目标缓冲区时。

大多数MPI库使用RDMA来实现小数据传输，因为它不需要接收方进程发布和处理任何IB请求。相反，发送方进程设置一个标志，作为写入暂存缓冲区的数据的一部分。由于缓冲区的使用是预先安排的，因此接收进程轮询相应的内存位置以检测新消息的到达。在主机缓冲区的情况下，内存轮询和内存复制都不会造成很大的开销。然而，当涉及到GPU缓冲区时，这些考虑因素会发生变化。在GPU内存上轮询CPU进程是非常昂贵的。此外，从暂存主机或GPU缓冲区到目标GPU缓冲区的额外副本会抵消或显著降低使用GDR的好处。可以通过使用IB Send/Recv操作而不是RDMA实现渴望协议来避免轮询开销，并支付较低的请求处理开销。但是，仍然需要额外的副本。我们考虑的另一个选择是使用IB提供的RDMA Immediate操作，这是一个在接收端生成完成的RDMA操作。这消除了对内存轮询的需要，但如果使用了暂存缓冲区，则仍然需要额外的副本。

由于这些限制，我们使用会合协议来实现使用GDR的所有消息传输。发送方和接收方进程在发送实际数据之前交换包含源/目的缓冲区的地址和注册信息的控制消息使用RDMA操作传输。使用GDR绕过发送方和接收方的主机内存的数据移动路径如图4所示。控制消息在主机内存之间交换，与从GPU内存移动数据相比，它们的开销很低。GPU内存的IB注册与主机内存的情况一样昂贵。因此，我们使用注册缓存来存储重用缓冲区的注册项，从而隐藏了这个开销。控制消息在主机内存之间交换，与从GPU内存移动数据相比，它们的开销很低。GPU内存的IB注册与主机内存的情况一样昂贵。因此，我们使用注册缓存来存储重用缓冲区的注册项，从而隐藏了这个开销。由于GPU设备内存分配是由CUDA库处理的，我们目前捕获cudaFree调用(或其驱动程序等效)，以确保在内存被释放之前内存未注册。释放和重新分配仍然在IB适配器上注册的CUDA设备内存会导致当前OFED堆栈中的未定义行为。对于可移植性来说，提供回调到MPI库或在释放任何已注册内存时自动取消注册的机制是可取的。我们希望这种机制将在GDR的未来版本中可用。

B. MVAPICH2 GPUDirect RDMA Hybrid (MV2-GDR-H)

为了克服在现代英特尔架构上使用GDR所带来的带宽限制，我们提出了一种介于MVAPICH2中现有的基于主机的管道解决方案和新的基于GDR的设计之间的混合解决方案。对于延迟很关键的小消息，我们使用快速GDR路径，对于大消息，我们切换回现有的基于主机的解决方案，它绕过了P2P读取限制。在这两种解决方案之间切换的阈值取决于底层平台的性能特征。我们将在第IV-B节中深入研究这个问题。

设计FpgaNIC

  1. 卸载控制平面到GPU如，让FPGA访问到GPU的虚拟地址

在NVIDIA GPUDirect的支持下，FPGA通过PCIe IP核中的DMA引擎可以发出DMA read/write命令，命令由起始物理地址和长度组成，其中长度不能大于一个GPU页大小，由此高效地与GPU双向传输数据。

但是存在两个问题：一是GPU程序使用的是GPU虚拟地址而不是物理地址；二是一块虚拟地址连续的地址空间物理地址并不连续，通常GPU的页大小为64KB，并且GPU还不支持大页内存，当数据块很大时，FPGA需要维护一个复杂的TLB。

因此设计了GPU通信栈，允许FPGA使用GPU虚拟地址直接访问GPU内存。GPU通信栈构建在PCIe IP核中，将控制平面卸载到GPU中，将数据平面卸载到FPGA中。

为了能够让GPU能够直接访问到FPGA的控制和状态寄存器，将控制平面卸载到GPU，需要采用软硬件协同设计的方法。在软件方面，我们的实现由GPU驱动程序、FPGA驱动程序和与这两个驱动程序交互的用户代码组成。首先，FPGA驱动程序使用misc_register函数将PCIe BAR作为IO设备注册到Linux内核中/dev/fpga_control。其次，用户代码使用mmap函数将设备映射到主机地址。第三，用户代码采用CUDA(计算统一设备架构)内存管理功能来注册主机地址，以便在CUDA内核中使用。这样，FPGA上的控制寄存器被映射到GPU的虚拟地址空间，GPU可以触发FPGA上的门铃寄存器和轮询状态寄存器，而无需CPU干预。并且，通过GPU通过写FPGA控制和状态寄存器，告诉FPGA GPU的虚拟地址到物理地址的映射关系。

UVA

统一虚拟地址(Unified Virtual Addressing, UVA)是CUDA 4.0提出的内存地址管理系统。当CUDA运行初始化时，会将应用程序的地址空间划分为两个部分：CUDA管理的虚拟地址范围、OS管理的虚拟地址范围。

PCI接口

PCI规范定义了计算机中各个组件之间如何互联。这篇文章主要理清PCI驱动如何找到硬件设备并访问它。

在PCI规范中，定义了一个三元组标识每个PCI外设的身份信息：(bus number, device number, function number)，但是在驱动中，使用的是pci_dev结构体来表示一个PCI设备。PCI设备实际上是一个树形结构，使用pciutil中的lspci命令可以查看系统中的PCI设备，/proc/pci和/proc/bus/pci文件也描述了PCI设备的信息。

物理存储介质

在大多数计算机系统中存在几种类型的存储介质，这些存储介质根据存取数据的速度、购买介质单位数据的成本以及介质的可靠性进行分类:

  • Cache - 高速缓存是最快和最昂贵的存储形式，缓存相对较小，由计算机系统硬件管理。
  • Main Memory - 主存储器存放可操作数据和机器指令。个人电脑的主存大概为几百GB大小，大型服务器系统主存可以达到数千GB大小。对于需要存储大量数据的数据库来说，主存通常太小(或太昂贵)，但随着主存的增大，许多企业数据库也可以放在主内存中。然而，主存作为易失性存储器，在电源故障或系统崩溃的情况下内容会丢失。
  • Flash Memory - 闪存作为非易失性存储器，即使电源关闭或发生故障，存储的数据也会被保留。闪存的每字节成本比主存低，但比磁盘的每字节成本高。闪存被广泛用于照相机和手机等设备的数据存储，闪存也用于在“USB闪存驱动器”中存储数据，即我们常用的“U盘”。随着价格的降低，在个人计算机中，用闪存来替代磁盘已经是很常见的事情。固态硬盘(solid-state disk, SSD)内部使用闪存存储数据的同时，提供类似于磁盘的接口，允许以块为单位存储或检索数据，这样的接口称为面向块的接口，块大小通常从512B到8KB节不等。
  • Magnetic-disk storage - 长期在线存储数据的主要介质是磁盘驱动器，也称为硬盘驱动器(hard-disk driver, HDD)。磁盘和闪存一样，是非易失性的。要访问存储在磁盘上的数据，系统首先将数据从磁盘移动到主存储器，从主存储器可以访问这些数据。当系统执行指定的操作后，需要将修改过的数据写入磁盘。尽管比ssd便宜得多，但就每秒支持的数据访问操作数量而言，磁盘的性能较低。

存储器接口

高速接口

磁盘和固态硬盘可以通过高速互连接口与计算机系统中的其他部件连接：

  • SATA(Serial ATA)接口 - SATA-3版本理论上支持6GB/s，允许数据传输速度高达600MB/s。
  • SAS(Serial Attached SCSI)接口 - SAS接口在服务器领域应用的更加广泛，SAS-3支持12GB/s的数据传输速率。
  • NVMe(Non-Volatile Memory Express) - 为了更好地支持SSD，2010年左右提出的NVMe接口标准，通常与PCIe接口一起使用。

SAN

当磁盘和固态硬盘与计算机系统中的其他部件的物理距离很远时，需要通过高速网络将磁盘连接到计算机系统。因此，提出了存储区域网络(storage area network, SAN)架构，大量的磁盘和数个服务器通过高速网络互连，这些磁盘通过独立磁盘的冗余阵列(redundant arrays of independent disks, RAID)技术为服务器提供逻辑上大而可靠的磁盘存储。SAN中，常见的网络互连技术：

  • iSCSI - 通过IP发送SCSI命令
  • FC(Fiber Channel) - 支持1.6~12GB/s的传输速率
  • InfiniBand - 低延时、高带宽网络

NAS

NAS(Network attached storage)是SAN的替代方案，它利用网络文件系统协议(NFS)或CIFS提供了文件系统接口。

闪存

与磁盘相比，SSD提供了更高的性能，但是
有两种类型的闪存：

  • NOR flash - NOR Flash和NAND Flash都是非易失性存储器，它们都可以用于嵌入式系统和移动设备等应用中。它们的工作原理和应用场景有所不同。

NOR Flash是一种串行存储器，它的存储单元可以直接寻址。它的读取速度比较快，因为数据可以直接从存储单元读取。NOR Flash还支持随机访问，因此适用于嵌入式系统中的代码存储和执行，例如固件存储和引导加载器等。NOR Flash的缺点是擦除速度比较慢，而且价格相对较高。

NAND Flash是一种并行存储器，它的存储单元不能直接寻址。它的读取速度比较慢，因为需要将数据从存储单元读取到缓冲区，然后再传输到CPU。但是，NAND Flash的擦除速度比较快，且价格相对较低。因此，NAND Flash适用于大容量数据存储，例如移动设备中的媒体文件和数据文件等。

总体来说，NOR Flash适合于代码存储和执行，而NAND Flash适合于数据存储和读取。在实际应用中，NOR Flash和NAND Flash也可以组合使用，以满足不同的存储需求。

Xilinx MPSoC启动流程

  1. 加载FSBL(First Stage Boot Loader) - 当设备上电或者复位时，硬件会自动从flash中加载FSBL，FSBL是一个预制的引导程序，它的作用是初始化硬件、设置时钟、检测系统状态等，并且加载bitstream文件到FPGA芯片中。
  2. 加载ATF(ARM Trusted Firmware) - FSBL加载bitstream后，会将控制权转移给ATF，ATF是由ARM维护的可信固件，它的作用是初始化CPU、设置内存、加载U-Boot等。
  3. 加载U-Boot - ATF加载完成后，会将控制权转移给U-Boot,U-Boot是一种开源启动程序，它的作用是初始化硬件、加载Linux内核和设备树等信息，并设置启动参数，最终将控制权转交给Linux内核。
  4. 加载Linux内核 - U-Boot将控制权转交给Linux内核，Linux内核是操作系统的核心，它的作用是完成系统初始化、加载驱动程序和应用程序等，最终启动Linux操作系统。

Xilinx提供了Bootgen工具，Bootgen可以将上述FSBL、ATF、U-Boot、Linux内核、设备树等文件打包生成启动镜像，最终将其写入flash等存储设备中。

PetaLinux

PetaLinux是Xilinx公司推出的一款基于Yocto Project 的嵌入式 Linux 开发工具，旨在简化嵌入式 Linux 系统的开发和定制。PetaLinux 提供了一套完整的工具链，包括交叉编译工具链、构建系统、软件包管理器、设备树编辑器等，可以方便地进行 Linux 内核和文件系统的构建、配置、调试和部署。
PetaLinux 支持多种硬件平台，其中包括 Xilinx 的 Zynq、Zynq UltraScale+ MPSoC 等 SoC 平台，以及 Virtex、Kintex、Artix 等 FPGA 平台。PetaLinux 可以与 Xilinx Vivado 集成，实现 FPGA 和 Linux 内核的联合设计和调试。
PetaLinux 的主要功能包括：

  1. 嵌入式 Linux 系统的构建和定制：PetaLinux 提供了一整套工具链，可以方便地构建和定制嵌入式 Linux 系统，包括 Linux 内核、文件系统、设备驱动等。

  2. 设备树编辑器：PetaLinux 集成了设备树编辑器，可以方便地编辑设备树文件，实现硬件和软件的匹配和配置。

  3. 软件包管理器：PetaLinux 提供了软件包管理器，可以方便地添加、移除和更新软件包，支持多种软件包格式和源。

  4. 交叉编译工具链：PetaLinux 提供了一套交叉编译工具链，支持多种处理器架构和操作系统，可以方便地进行开发和调试。

  5. Vivado 集成：PetaLinux 可以与 Xilinx Vivado 集成，实现 FPGA 和 Linux 内核的联合设计和调试，提高开发效率。

Petalinux打包镜像

petalinux-build命令用于编译PetaLinux项目，会生成多个文件和目录：

  • images/linux目录 - 包含了生成的Linux内核、设备树和启动镜像等文件。
  • components/plnx_workspace目录：包含了编译过程中的中间文件和工作区。
  • build目录：包含了编译过程中生成的编译日志和其他文件。
  • rootfs目录：包含了根文件系统的文件和目录，用于构建 Linux 系统的根文件系统。
  • project-spec目录：包含了 PetaLinux 项目的配置文件和自定义组件等信息。

petalinux-package --boot命令用于将PetaLinux项目打包成启动镜像，可以烧写到闪存或SD卡中启动设备。该命令需要指定以下参数：

  • –fsbl - 指定FSBL文件的路径，FSBL是Zynq MPSoC设备启动的第一个阶段，用于初始化硬件和加载bitstream。
  • –fpga - 指定bitstream文件的路径，如果使用了FPGA，则需要将bitstream文件加载到FPGA中。
  • –u-boot - 指定U-Boot文件的路径，U-Boot是设备启动的第二个阶段，在FSBL启动完成后被加载并执行。
  • –force - 强制覆盖已存在的镜像文件。

Xilinx SDK打包镜像

  • zynqmp_fsbl.elf: 是Xilinx Zynq MPSoC平台上的(FSBL)文件，它是启动Zynq MPSoC设备时第一个被执行的程序。FSBL主要工作是初始化硬件系统，加载bitstream(如果使用可重构逻辑FPGA的话)，执行一些基本的安全性检查，最后跳转到第二阶段的u-boot或者操作系统的启动程序。
  • pmu.elf: 用于在ARM架构的处理器上启用性能计数器和性能分析功能。PMU代表性能监控单元，它是ARM架构中的一个硬件模块，用于收集和分析处理器的性能数据。pmu.elf文件包含了一组用于配置和控制PMU的指令，以及一些用于读取和分析性能计数器数据的工具。在处理器上运行pmu.elf文件可以启用性能计数器和性能分析功能，并收集处理器的各种性能数据，例如指令执行次数、缓存命中率、分支预测成功率等等。

在PetaLinux开发流程中，zynqmp_fsbl.elf和pmu.elf是通过Xilinx SDK工具链编译生成的，在PetaLinux项目中通常位于<project>/images/linux目录下。

  1. 打开Xilinx SDK，新建一个workspace，选择File->New->Application Project，在弹出的对话框中的Hardware Platform中选择New，选择HDF创建新的硬件平台。

  2. 创建FSBL：配置好之后，选择Next->ZynqMP FSBL，Finish，通过fsbl_bsp->system.mss更改BSP属性。

  3. 创建PMU：使用第一步创建的硬件平台，通过fsbl_bsp->system.mss更改BSP属性。

  4. 打包BOOT.bin：由fsbl.elf、pmu.elf、bit、bl.elf、u-boot.elf、image.ub组成，选择Xilinx->Create Boot Image导入。

  5. 烧入FLASH：选择Xilinx->Program Flash，正常烧入需要30分钟。
